Beginner — Week 1–2

Foundation Layer

Installation & Project Setup
npm init playwright@latest, tsconfig.json, playwright.config.ts anatomy
Why: Every test run starts here
Mistake: Skipping tsconfig strict mode — causes type errors later you can't explain
Core Locator Strategies
getByRole, getByLabel, getByText, getByTestId, locator() — priority order matters
Why: 80% of flakiness comes from bad locators
Mistake: Using CSS selectors or XPath as first choice — they break on redesigns
Actions & Assertions
click, fill, check, selectOption, expect().toBeVisible(), toHaveText(), toHaveValue()
Why: Core building blocks of every test
Mistake: Using expect(await page.textContent()...) instead of auto-retrying expect(locator)
Auto-Waiting Internals
How Playwright waits for actionability — enabled, visible, stable, receives events
Why: Eliminates 90% of sleep() anti-patterns
Mistake: Adding page.waitForTimeout() everywhere — this is a crutch that masks real issues
Test Structure & Lifecycle
test(), test.describe(), beforeEach, afterAll, test.use(), test.step()
Why: Clean structure = readable, maintainable tests
Intermediate — Week 3–4

Professional Layer

Page Object Model (POM)
Encapsulate selectors and actions per page, typed interfaces, composition patterns
Why: Scale past 50 tests without chaos
Mistake: Putting assertions inside page objects — keep them in tests only
Fixtures (Playwright's superpower)
test.extend(), scoping (test/worker), fixture composition, authenticated states
Why: DRY setup/teardown, shareable across test files
Pro tip: Replace beforeEach blocks with fixtures — they compose better and are typed
API Testing with Playwright
request context, apiRequestContext, seeding data via API before UI tests
Why: API setup is 10x faster than UI setup for test data
Network Interception
page.route(), fulfill(), abort(), mock APIs, intercept and assert request payloads
Why: Test error states without breaking the backend
Authentication Patterns
storageState, globalSetup login, per-role auth fixtures, avoiding re-login overhead
Why: Without this, your suite is 3x slower
Parallel Execution
workers config, test.describe.parallel, sharding across machines, worker-level isolation
Why: 500 tests must run in under 5 minutes in CI

Learning Philosophy

Rule #1: Write code within 30 minutes of reading any concept. No code = no retention.
Mini-Task: Locator Dojo
Go to playwright.dev itself and write 10 tests using ONLY getByRole/getByLabel. No CSS allowed.
Readiness: 0 CSS selectors in your tests
Mini-Task: Auto-wait Proof
Build a test with a 2s delayed element. Prove auto-wait works. Then break it with a bad locator to see the error.
Readiness: You can explain actionability without notes
Mini-Task: POM Build
Pick any free site (SauceDemo, OrangeHRM). Build 3 page objects. Write 5 tests using them. Refactor until tests have ZERO raw locators.
Readiness: Test file imports POM only — no page.locator() in tests
Mini-Task: Fixture Ladder
Convert 3 beforeEach blocks to fixtures. Add a worker-scoped DB seed fixture. Measure time saved.
Readiness: You can explain scope difference (test vs worker) verbally
Mini-Task: API + UI Chain
Create a user via API → log in via UI → assert profile data. No UI form filling for setup.
Readiness: Setup is 100% via API, only assertions are UI
Mini-Task: Network Mock Lab
Mock a failed API call (500 error). Assert UI shows the correct error state. Never touch the backend.
Readiness: You can test any error state without backend cooperation
Mini-Task: Parallel Speed Test
Run 20 tests with workers:1 then workers:4. Record times. Tune for your machine. Add sharding config.
Readiness: Suite runs 3x faster with parallel config
Mini-Task: Trace Archaeology
Deliberately fail a test. Open trace. Navigate action-by-action. Find the exact failure point. Fix it using trace only (no console.log).
Readiness: You never use console.log to debug Playwright

POM vs Fixtures vs Hybrid

POM Only
Good for: teams new to Playwright, simple apps

Drawback: setup/teardown still duplicated in beforeEach blocks
Fixtures Only
Good for: API testing, auth state, DB seeding

Drawback: heavy UI interactions still need page abstractions
Hybrid (Use This)
POM for page abstractions + Fixtures for setup/teardown/auth.

Industry standard for production frameworks

Test Data Management

Static data: JSON files in /test-data — users, products, configs. Versioned in git. Good for stable reference data.
Dynamic data: DataFactory.ts — generates unique test data per run (faker.js). Prevents test collision in parallel runs.
API seeding: Create data via API in fixture beforeAll. Tear down in afterAll. Zero UI dependency for setup.
Environment isolation: Each parallel worker gets its own test data namespace (WORKER_ID prefix on usernames).

Environment Config Pattern

// EnvConfig.ts import { PlaywrightTestConfig } from '@playwright/test'; const ENV = process.env.TEST_ENV || 'staging'; const envConfig = { local: { baseURL: 'http://localhost:3000', apiURL: 'http://localhost:8080' }, staging: { baseURL: 'https://staging.drcinsight.com', apiURL: 'https://api-staging.drc.com' }, prod: { baseURL: 'https://drcinsight.com', apiURL: 'https://api.drc.com' }, }; export const config = envConfig[ENV]; // playwright.config.ts — use: npx playwright test --env=staging export default { use: { baseURL: config.baseURL } };

Reporting Stack

HTML Reporter: Built-in. Perfect for local dev. Add to .gitignore. Run: npx playwright show-report
Allure Reporter: allure-playwright npm package. Rich dashboards, trends, categories. Best for CI pipelines shared with team.
JSON Reporter: Use for custom downstream processing — post results to Jira, Slack, dashboards.
Custom Reporter: Implement the Reporter interface. onTestEnd() → post Slack webhook. onEnd() → generate Jira ticket for failures.

Dockerfile — Production-Grade

FROM mcr.microsoft.com/playwright:v1.44.0-jammy WORKDIR /app # Copy dependencies first (layer caching) COPY package*.json ./ RUN npm ci # Copy source COPY . . # Run as non-root user (security) RUN groupadd -r playwright && useradd -r -g playwright playwright USER playwright # Default: run full suite CMD ["npx", "playwright", "test"]
Use mcr.microsoft.com/playwright official image — browsers pre-installed. Never RUN npx playwright install in CI — it's slow and unreliable.

GitHub Actions — PR Gate (Smoke)

name: Playwright Smoke Tests on: pull_request: branches: [main, develop] jobs: smoke-tests: runs-on: ubuntu-latest container: image: mcr.microsoft.com/playwright:v1.44.0-jammy steps: - uses: actions/checkout@v4 - name: Install deps run: npm ci - name: Run smoke tests run: npx playwright test --grep @smoke env: TEST_ENV: staging BASE_URL: ${{ secrets.STAGING_URL }} API_KEY: ${{ secrets.API_KEY }} - name: Upload report if: always() uses: actions/upload-artifact@v4 with: name: playwright-report path: playwright-report/ retention-days: 30

GitHub Actions — Nightly Regression with Sharding

name: Nightly Regression on: schedule: - cron: '0 2 * * *' # 2 AM daily jobs: test: runs-on: ubuntu-latest container: image: mcr.microsoft.com/playwright:v1.44.0-jammy strategy: fail-fast: false matrix: shard: [1, 2, 3, 4] # 4 parallel shards steps: - uses: actions/checkout@v4 - run: npm ci - name: Run shard run: npx playwright test --shard=${{ matrix.shard }}/4 env: TEST_ENV: staging - uses: actions/upload-artifact@v4 if: always() with: name: blob-report-${{ matrix.shard }} path: blob-report/ merge-reports: needs: test runs-on: ubuntu-latest steps: - uses: actions/checkout@v4 - run: npm ci - uses: actions/download-artifact@v4 with: { pattern: blob-report-*, merge-multiple: true, path: all-blobs/ } - run: npx playwright merge-reports --reporter html ./all-blobs - uses: actions/upload-artifact@v4 with: { name: final-report, path: playwright-report/ }

Flaky Test Handling in CI

retries: 2 in playwright.config — only retries in CI (set retries: process.env.CI ? 2 : 0)
Quarantine tag: mark known flaky as @flaky, exclude from PR gate: --grep-invert @flaky
trace: 'on-first-retry' — traces only generated when a test fails once, keeps CI storage clean
Flaky report: parse JSON output in CI, post any test with retries > 0 to Slack as a flakiness alert

Smart Locator Strategy — Priority Order

1st — getByRole() · Semantic, accessibility-aligned, change-resistant
2nd — getByLabel() · Form fields. Never fails on input type changes
3rd — getByText() / getByPlaceholder() · For static text you control
4th — data-testid · When devs add them. Better than CSS. Never use auto-generated IDs.
Avoid — CSS classes, XPath, nth-child, auto-generated class names

Flaky Test Prevention

Never use page.waitForTimeout()
Use explicit assertions instead. expect(locator).toBeVisible() already retries. If you need to wait, your locator is wrong or your assertion is wrong.
Isolate test state completely
Each test must create its own data and clean up. Never share mutable state between tests. Parallel tests that share data will collide randomly.
Avoid testing timing-dependent behavior
Use network interception to control API response speed. Never assert "animation complete" — assert the final state only.
Use strict mode locators
Enable strict:true in config. Forces you to use unique locators. Multiple matches = test error. This catches bad locators at authoring time, not in CI.
Scope locators to their component
const modal = page.getByRole('dialog'); modal.getByRole('button', {name:'Save'}) — not page.getByRole('button', {name:'Save'}). Pages have many Save buttons.

Code Review Checklist

No raw CSS selectors or XPath in test files
No page.waitForTimeout() anywhere in the codebase
No assertions inside Page Objects
Every test is independent — can run alone or in any order
Sensitive data (passwords, tokens) from env vars only — never hardcoded
Test has a clear Arrange / Act / Assert structure
Test name describes what it proves, not what it does
New test has a @smoke or @regression tag

How Top Engineers Learn

Build first, polish later
Working ugly code is worth more than perfect notes. Get something running in under 30 minutes of starting any new topic. Refactor later. Learning happens in the doing.
Read the actual source code
Open node_modules/@playwright/test/src in VSCode. Read how auto-wait works. How fixtures chain. You'll understand it 10x better than any tutorial.
Teach it back
After each week, explain what you learned as if coaching Andy or Jordon. If you stumble, that's exactly what you need to review. Write a short internal wiki.
Escape tutorial hell
Limit YouTube to 1 hour per topic maximum. The moment you understand the concept, stop the video and build it yourself. Watching a second tutorial on the same topic = wasted time.

Think Like a Test Architect

Ask "what breaks in parallel?" before writing any test
Architects think in scale from day one. Every test you write: can 10 workers run this simultaneously without conflict? If not, fix the data isolation first.
Treat your framework as a product
Write a README. Version it. Make it installable. Document the fixture API like it's a public SDK. This mindset produces frameworks other engineers actually use.
Instrument everything
Capture: test duration, retry count, environment, build ID per run. You need this data to argue for infrastructure investment and prove ROI to management.
Design for the junior who joins in 6 months
If they can't write their first test in 30 minutes using your framework, it's too complex. The best frameworks constrain the wrong choices automatically.
